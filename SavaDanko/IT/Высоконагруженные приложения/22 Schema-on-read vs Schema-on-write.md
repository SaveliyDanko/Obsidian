###### **Что значит "бессхемная база данных"?**
Когда говорят, что база "бессхемная" (schemaless), это не означает, что данные в ней хаотичны или неструктурированы. Это значит, что база данных сама не требует заранее определённой структуры (схемы) для хранения данных.

Например, в MongoDB ты можешь записать в одну коллекцию такие документы:
```json
{ "name": "Alice", "age": 25 }
{ "username": "Bob42", "isActive": true }
```
И база не выдаст ошибку, хотя структуры у документов разные.

###### **Но как тогда код работает с такими данными?**
Тут важно понимать: хотя база не требует схемы, программист почти всегда её предполагает. Когда ты читаешь документ из базы, ты обычно ожидаешь, что он содержит определённые поля — например, `"name"` или `"age"`. Если этих полей нет или они не того типа — код сломается.

То есть схема всё равно есть, просто она не навязывается базой, а существует в голове разработчика или в коде приложения.

###### **Schema-on-read vs. Schema-on-write**
Вот здесь вводится важное различие:

| Подход          | Название           | Как работает?                                                                                                       |
| --------------- | ------------------ | ------------------------------------------------------------------------------------------------------------------- |
| Schema-on-write | «Схема при записи» | Сначала задаётся схема (структура), и только потом можно писать данные. База сама проверяет, что всё соответствует. |
| Schema-on-read  | «Схема при чтении» | Данные пишутся как есть. Схема применяется только при чтении, в коде, который эти данные обрабатывает.              |
