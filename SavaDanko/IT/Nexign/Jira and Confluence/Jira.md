###### **Основные шаги**
1. После встречи создаются два типа задач:
    - Improvement — протокол встречи.
    - New Feature — новая функциональность, которую нужно реализовать.  
        (если требований несколько, создаются несколько задач).

Таким образом, протокол встречи в JIRA фиксируется как Improvement, а все требования переводятся в New Feature, которые при необходимости дополняются задачами на тестирование и разработку.

###### **JIRA-issue типа Improvement**
- Improvement фиксирует исходное требование заказчика (бизнес-описание).
- Создаётся заказчиком или сотрудником Nexign (по договорённости).
- Используется для согласования требований внутри команды заказчика.
- После утверждения заказчиком внутри своей команды:
    - требование считается сформированным
    - Issue переводится в статус Resolved
    - передаётся в Nexign.

###### **Какие поля заполняются**
При создании Improvement указываются:
- Project — код проекта
- Issue Type — Improvement
- Summary — краткое название требования
- Description — описание требования
- Assignee — назначенное лицо
- Person Responsible on PS Side — ответственный со стороны Nexign
- Person Responsible on Customer Side — ответственный со стороны заказчика
- Segment — в зависимости от требования
- Group and Subgroup — в зависимости от требования
- Priority — по умолчанию Medium (может быть изменён)
- Все остальные поля остаются пустыми (по умолчанию).

###### **Связь с New Feature**
1. При анализе Improvement формируется очередь требований.
2. Для каждой страницы ТЗ создаётся New Feature в статусе Open.
3. Каждая New Feature обязательно связывается с учтёнными Improvement через связь: New Feature → relates to → Improvement

###### **JIRA-issue типа New Feature**
- Создаётся сотрудником NX на основе требований заказчика.
- Используется для анализа и уточнения описания.
- После подготовки данных фиксируется и отправляется заказчику на согласование.
    
###### **Процесс согласования**
1. NX берёт Issue в работу → переводит Open → In progress.
2. В Assignee и Person Responsible on PS Side указывает себя.
3. При уточнении деталей идёт переписка с заказчиком (In progress ↔ Request to customer).
4. После внесения всех данных NX переводит Issue в Resolved и назначает ответственного от заказчика.
    
5. Заказчик:
    - если согласен → переводит в Approved,
    - если не согласен → Reopened, добавляет комментарии.
        
###### **Поля при создании**
- Project — код проекта
- Issue Type — New Feature
- Summary — формулировка требования (можно из Improvement)
- Description — описание из Improvement
- Priority — Unprioritized
- Assignee — себя
- Person Responsible on PS Side — себя
- CC — коллеги проекта, если затрагивает несколько продуктов
- Person Responsible on Customer Side — ответственный со стороны заказчика (если нет — РП заказчика)
- Requirement Classification — Functional
- Requirement Type — Functional requirement
- New or Change Request — New (указать дату)
- Segment / Group and Subgroup — по требованию
- Component/s — соответствующий компонент (например, BUSINESS-PROCESS)
- Check at migration — галочка при необходимости
- Attachment — материалы (если есть)
- Linked Issues — связь с Improvement
- Issue — список Improvements
- Epic Link — ссылка на epic проекта


###### **Статусы New Feature**

| Статус              | Описание                                                             |
| ------------------- | -------------------------------------------------------------------- |
| Open                | Очередь задач на стороне NX, работа ещё не начата.                   |
| In Progress         | Выполнение работ по описанию бизнес-процесса в NX.                   |
| Request to customer | Не используется.                                                     |
| Resolved            | Описание бизнес-процесса подготовлено и ждёт согласования заказчика. |
| Approved            | Работа завершена, результат согласован заказчиком.                   |
| Design              | Не используется, согласованное Issue закрывается.                    |
| Design done         | Не используется, согласованное Issue закрывается.                    |
| Reopen              | Требуется внести изменения по мнению заказчика (возврат в работу).   |
| Postpone            | Работы приостановлены; может быть выставлен на любом этапе.          |
| Closed              | Работа полностью завершена.                                          |

###### **JIRA-issue типа Task**
- Создаётся для каждого согласованного требования (New Feature в статусе Approved).
- При этом:
    - New Feature переводится в статус Design,
    - создаётся Task с ссылкой realizes на New Feature.
- Таким образом, Task реализует требование, зафиксированное в New Feature.

###### **Заполнение полей**
- Project — код проекта
- Issue Type — Task
- Summary — как у New Feature
- Description
    - пустое, если предполагается конфигурирование,
    - если конфигурирование не требуется, нужно явно указать,
    - если ожидается доработка — зафиксировать, что требуется обновление к выходу доработки.
- Priority — Unprioritized
- Assignee — себя
- Person Responsible on PS Side — себя
- Linked Issues — указано в New Feature
- Epic Link — Solution setup (ссылка на Epic для setup specification)


---
###### **JIRA-issue типа Sub-Task**
- Используется, если требование затрагивает настройки в нескольких продуктах.
- Ответственный за головной Task создаёт необходимое количество Sub-Task для настройки.
- Конфигурирование всегда выполняется в рамках дочерних Issue, поэтому хотя бы один Sub-Task создаётся в любом случае.

###### **Заполнение полей**
- Project — код проекта
- Issue Type — Sub-task
- Summary — как у New Feature + название продукта (в скобках)
- Description — что именно нужно настроить
- Priority — Unprioritized
- Assignee — Automatic
- Component/s — соответствующий продукт; если относится к пользовательскому коду, то дополнительно CUSTOM

###### **Статусы Task/Sub-Task**

| Статус              | Описание                                                                                           |
| ------------------- | -------------------------------------------------------------------------------------------------- |
| Open                | Очередь требований на стороне NX, работа ещё не начата.                                            |
| In Progress         | Выполнение работ над требованием в NX.                                                             |
| Request to customer | Выполнение работ над требованием представителем заказчика.                                         |
| Resolved            | ТЗ подготовлено и ждёт согласования со стороны заказчика.                                          |
| Done                | Работа над Issue завершена, результат согласован обеими сторонами.                                 |
| Reopen              | Требование возвращено с замечаниями от заказчика. При повторном взятии NX переводит в In progress. |
| Postpone            | Работы приостановлены; статус может быть выставлен на любом этапе.                                 |
| Closed              | Работа окончательно завершена.                                                                     |

###### **JIRA-issue типа Testing**
- Для каждого New Feature создаётся отдельный Issue типа Testing.
- Даже если тестирование не требуется, всё равно создаётся Testing с подтверждением от заказчика.

###### **Процесс работы**
1. После создания содержимое Testing согласуется с заказчиком.    
2. Для завершённого Testing:
    - Issue переводится в Resolved
    - формируется нота со ссылкой на Test Suite,
    - назначается заказчику (Assignee = Person Responsible on Customer Side, Resolution = Done),
    - отправляется на согласование.
3. После согласования заказчиком Testing переводится в Approved.
4. UAT, связанные с Approved Testing, могут выполняться командой тестирования заказчика.
    - Для этого Test Suite назначается ответственному от заказчика.

###### **Поля JIRA-issue типа Testing**
- Project — код проекта
- Issue Type — Testing
- Summary — `TEST + как у New Feature + stage 2 (если второй этап тестов)`
    - пример: `TEST - <код проекта> - номер + текст + [(stage2)]`
- Description — пустое
- Priority — Unprioritized
- Assignee — ответственный от NX для New Feature
- CC — DA + коллеги по направлениям проекта (если Issue затрагивает несколько продуктов)
- Person Responsible on Customer Side — ответственный со стороны заказчика для New Feature
- Person Responsible on PS Side — ответственный от NX для New Feature
- Component/s — TESTRAIL
- Linked issue — связь с New Feature


###### **Статусы Testing**

| Статус          | Описание                                                          |
| --------------- | ----------------------------------------------------------------- |
| Open            | Очередь требований на стороне Nexign — тесты ещё не созданы.      |
| Creating a test | Ответственный от Nexign описывает тесты в TestRail.               |
| Resolved        | Тесты описаны NX, ожидается согласование заказчика.               |
| Approved        | Работа завершена, результат согласован обеими сторонами.          |
| Quality control | Группа/специалист QC проверяет тесты на соответствие требованиям. |
| Testing         | Выполняется тестирование функциональности по тестам из TestRail.  |
| Fix Bugs        | В процессе тестирования обнаружены ошибки; идёт исправление.      |
| Done            | Тестирование успешно завершено.                                   |
| Postpone        | Работы приостановлены; может быть выставлен на любом этапе.       |
| Closed          | Работа окончательно завершена.                                    |

###### **SRS: Создание страницы (фиксация требования)**
- Для каждого New Feature создаётся отдельная страница в Confluence.
- Название страницы = номер JIRA Issue + summary.
    - Пример: `Chinguitel-18 Currency`.

###### **Размещение**
- Страница требования размещается в разделе 4.1 System General Requirements (блок функциональных требований).    
- Конкретное место внутри раздела определяется содержанием требования. 
- Возможен перенос в другие разделы документа — уточнять у DA.

###### **Статусы**
- На момент создания страница фиксируется при статусе In progress.    
- В Approved Issue переводит заказчик, когда убедится, что требование корректно отражено в Confluence.

###### **Содержимое страницы**
- Обязательный элемент: Description.
- Дополнительно можно размещать:
    - блок-схемы,
    - рисунки,
    - ссылки на другие Issue,
    - таблицы,
    - прочие элементы.

###### **Требования в Confluence**
- Текст требования согласуется с заказчиком и после согласования не меняется.
- Функциональное требование должно соответствовать описанию в System requirements specification и быть проверено в ходе UAT перед сдачей системы (если не ждём доработку CLM).
    
- Важно:
    - не писать лишнего
    - чётко описывать только ту функциональность, которая действительно будет настраиваться
    - учитывать договорённости по ограничениям (см. High-Level Requirements Document).
- Если связано с CLM — описывается постановка от ДРПО.
- Если в System requirements specification чего-то не написано и нет отдельного требования, считается, что система должна работать «из коробки» (out-of-box).
- Детали настройки описываются отдельно    

###### **SSS: Создание страницы (описание настройки требования)**
- После того как Issue с типом New Feature переведено заказчиком в Approved.
- На странице фиксируется описание настроек, которые нужно выполнить в системах для поддержки требований из Issue.

###### **Документ**
- Используется отдельный документ Solution Setup Specification (SSS).
- По структуре документ дублирует главу 4 REQUIREMENTS из TRD (Technical Requirements Document).

###### **Пример структуры**
- General Information
- System Creation Purpose and Goals
- Overview of the Automation Object
- Requirements
    - System General Requirements   
    - Requirements to the Functions Implemented by the System
        - Billing and finances
            - Chinguitel-18 Currency

###### **Названия разделов**
- В Confluence страницы внутри одного проекта не могут иметь одинаковое название.
- Поэтому для разделов SSS добавляется префикс "Setup for".
    - Пример:
        - В SRS раздел: Billing and finances
        - В SSS раздел: Setup for Billing and finances

###### **Содержимое страницы**
1. В начале страницы добавляется макрос JIRA Issue/Filter (Single Issue):
    - Ссылка на созданный Task для настройки.
    - В настройках отображения (Display options) ничего менять не нужно.
        
2. Далее — описание необходимых настроек продукта в свободной форме:
    - графика
    - скриншоты
    - текст
    - таблицы
    - и другие материалы
