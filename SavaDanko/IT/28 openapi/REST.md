**REST (Representational State Transfer)** — это архитектурный стиль для построения распределённых систем, главным образом веб-сервисов.

Его предложил Рой Филдинг в своей докторской диссертации (2000).  
Идея REST — сделать взаимодействие между клиентом и сервером простым, масштабируемым и независимым от технологий.

В основе REST:
- ресурсы (entities), которые имеют URI (идентификатор)
- стандартные HTTP-методы для работы с ними (`GET`, `POST`, `PUT`, `PATCH`, `DELETE`)
- стандартные коды состояния для ответа (`200`, `201`, `404`, `500` и др.)
    

#### 6 ограничений REST
###### 1. Клиент-сервер (Client–Server)
- Разделение клиента (UI, фронтенд) и сервера (данные, логика).
- Клиент отвечает за интерфейс, сервер — за хранение и обработку ресурсов.
- Даёт гибкость и независимость (можно менять клиент или сервер отдельно).

###### 2. Отсутствие состояния (Stateless)
- Каждый запрос должен содержать всю необходимую информацию.
- Сервер не хранит состояние сессии между запросами.
- Если нужно отслеживать пользователя → используются токены (JWT, OAuth).
    
###### 3. Кэшируемость (Cacheable)
- Ответы сервера должны явно указывать, можно ли их кэшировать (`Cache-Control`, `ETag`).
- Это снижает нагрузку и повышает скорость.
- Пример: `GET /articles` можно кэшировать, а `POST /articles` — нет.
    
###### 4. Единообразный интерфейс (Uniform Interface)
Главное ограничение REST. Оно делает API простым и понятным.  
Состоит из четырёх принципов:
- Идентификация ресурсов через URI.
- Манипуляция ресурсами через представления (JSON, XML и др.).
- Самоописательные сообщения (достаточно заголовков + тела, чтобы понять смысл).
- Гипермедиа как средство управления состоянием (HATEOAS — ссылки внутри ответа для навигации по API).

###### 5. Многоуровневая система (Layered System)
- Клиент не должен знать, напрямую ли он общается с сервером или через прокси/балансировщик/кэш.
- Это упрощает масштабирование и безопасность.

###### 6. Код по требованию (Code on Demand, опционально)
- Сервер может передавать клиенту исполняемый код (например, JavaScript), чтобы расширить функционал.
